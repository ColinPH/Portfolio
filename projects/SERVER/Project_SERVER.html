This project exists because I wanted to replicate/use/improve some of the tools I saw when working in the industry. I have always been passionate about work pipelines and how tools and automation could support a team. I see this as my opportunity to learn, practice, explore, refine, test all the pipeline ideas I come up with.

I started this project with some theoretical knowledge but no practical experience in system administration. This has been a great opportunity to learn using online resources and ChatGPT. My approach with using ChatGPT has been and still is that I make sure I understand what I write before using it. And I heavily test what I learned to make sure I understand it on a deeper level.

Server Assembly

Having used laptops, I always wanted to assemble a computer. This was the perfect opportunity. I used a second-hand motherboard with CPU, RAM and cooler. Around which I built the other components.

The server does not have a GPU installed. This was deliberate because of energy consumption and due to the fact that the Unity builds I intended to run required only the CPU. I have plans on adding a GPU when project builds require it. Or when I integrate some automated test that run the executable.

The server has multiple drives with each their own responsibility. An NVME for running the software, an SSD for storing the projects and repos. And a HDD for storing backups.

Git Repos

On the server, I used Git to host my own repositories.

Having setup multiple repos for my past projects using github or Plastic SCM. I wanted to know how to host my own repo. And through that being able to own my data and not have to rely on paid subscriptions.

Jenkins

The server runs Jenkins to manage the different automated tasks and manual publications.

Jenkins dashboard with an overview of all the projects and pipelines.

Most pipelines are automatic. But some pipelines are made for the users to interact with the server. Whether that is to publish a given build, or remove a build from the archive.

Parameters for manually publishing a build to platforms.

Package Registry

I use Verdaccio to host a upm package registry. This is a way for me and my team to create packages and share them across projects using the Package Manager in Unity. This makes it much easier to control what packages we want in each project.

You can learn more about the packages I host in the FRAMEWORK Project.

Web interface of the Verdaccio package registry.

Discord Bot

The server runs a discord bot that informs the team, with some humor, about the state of the automated build and publication. When the message is posted it means that people from the team can download the build from the server.

Message posted by the discord bot when the build has been successful.

If a build fails because of some compilation errors, the bot provides a file with only the relevant errors. Just like we would see them in the Unity console when building. This has been done so that the programmers can easily see the errors and start working on them without having to fetch them from the server manually.

Message posted by the discord bot when the build has failed.

One of the Jenkins pipelines allows users to publish a specific build to the desired platform. The discord bot also announces this to help the team stay informed about what is happening.

Message posted by the discord bot when a build has been published.

Samba File Share

To provide the team with a place to share files and data, I have set up a Samba share on the server. This allows us to easily access the shared directories from our machine explorer windows. While not having to worry about any fees or other limitations that can occur with solutions like Google Drive.
